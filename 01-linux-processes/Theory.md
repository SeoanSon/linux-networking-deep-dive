# Theory: Linux 프로세스와 IPC의 기초

## 1️⃣ 프로세스란 무엇인가?

### 정의

**프로세스** = 실행 중인 프로그램의 인스턴스 + 실행 환경

```
프로그램 (정적) → 프로세스 (동적)
   /bin/bash      PID 1000인 bash 인스턴스
   python         PID 1001인 python 인스턴스
```

### 프로세스의 고유한 특성

1. **PID (Process ID)** - 커널이 부여하는 고유 식별자
2. **메모리 공간** - 다른 프로세스와 완전히 격리
3. **파일 디스크립터 테이블** - 열어놓은 파일 목록
4. **권한 (UID/GID)** - 누가 실행하는가
5. **상태** - Running, Sleeping, Stopped, Zombie

### 아키텍처: 프로세스 메모리 레이아웃

```
┌─────────────────────────────────────┐
│  커널 공간 (모든 프로세스 공유)        │
│  - 프로세스 테이블                   │
│  - 인터럽트 핸들러                   │
│  - 드라이버 코드                     │
└─────────────────────────────────────┘
         ↑ 3GB 경계 (32-bit)
┌─────────────────────────────────────┐
│  사용자 공간 (프로세스 A)             │
├─────────────────────────────────────┤
│  스택 (Automatic)                   │ ← 로컬 변수
│  ↑                                  │
│  (동적 증가)                         │
│                                     │
│  (미할당 영역)                       │
│                                     │
│  ↓                                  │
│  힙 (Malloc)                        │ ← 동적 할당
├─────────────────────────────────────┤
│  .bss (Uninitialized data)          │
│  .data (Initialized data)           │
│  .text (코드)                       │
└─────────────────────────────────────┘
```

**중요**: 각 프로세스는 자신의 **Virtual Memory** 주소공간을 가짐
- 프로세스 A의 0x400000 ≠ 프로세스 B의 0x400000
- MMU (Memory Management Unit)가 물리 메모리로 변환

## 2️⃣ 프로세스 격리: 왜 필요한가?

### 문제 상황 (격리 없이)

```
프로세스 A: 버그 → 메모리 오버플로우 → 프로세스 B의 메모리 덮어쓰기
결과: 프로세스 B가 예기치 않게 충돌
```

### 해결책: 격리 메커니즘

**1. 메모리 격리**
- 각 프로세스가 자신만의 Virtual Memory 주소공간 사용
- 프로세스 A가 프로세스 B의 메모리에 접근 불가능

**2. 파일 디스크립터 격리**
- 각 프로세스가 자신의 파일 테이블 유지
- 프로세스 A가 연 파일과 프로세스 B가 연 파일은 분리

**3. 권한 기반 격리 (UID/GID)**
- 프로세스 A (UID=1000)가 프로세스 B (UID=2000)의 리소스 접근 불가
- 예외: root (UID=0)는 모든 리소스 접근 가능

**4. 신호 기반 격리**
- 다른 권한의 프로세스에게 신호 전송 불가
- 예외: 부모는 자식 프로세스 제어 가능

## 3️⃣ UID/GID: 프로세스의 신원

### UID (User ID)

```bash
$ id
uid=1000(user) gid=1000(user) groups=1000(user)
```

**의미**:
- UID 1000: 현재 사용자의 신원
- 모든 프로세스는 실행 시 UID를 가짐
- 커널이 UID로 권한 확인

### setuid 비트: 중요한 보안 메커니즘

```
-rwsr-xr-x 1 root root /usr/bin/passwd
```

맨 앞의 's' = setuid 비트
- passwd 명령어를 root가 소유
- **일반 사용자가 실행해도 root 권한으로 실행됨**
- 따라서 사용자가 자신의 비밀번호 변경 가능

```bash
$ whoami
user

$ passwd
# 위 명령어 실행 시
# 프로세스는 UID=1000이 아닌 UID=0 (root)으로 실행됨!
```

## 4️⃣ 프로세스 생명주기

### 상태 전이

```
fork() 생성
   ↓
┌──────────────┐
│   Running    │ ← CPU에서 실행 중
│ (R 상태)    │
└──────────────┘
   ↓ (I/O 대기)
┌──────────────┐
│   Sleeping   │ ← I/O 대기 중
│ (S 상태)    │
└──────────────┘
   ↓ (깨어남)
┌──────────────┐
│   Running    │
└──────────────┘
   ↓ (exit())
┌──────────────┐
│   Zombie     │ ← 종료했으나 부모가 처리 안 함
│ (Z 상태)    │
└──────────────┘
   ↓ (wait())
   소멸
```

### Zombie 프로세스의 위험성

```c
// 부모 프로세스
if (fork() == 0) {
    // 자식 프로세스
    exit(0);
}
// 부모가 wait() 호출 안 함!
sleep(1000); // 계속 실행
```

결과: 자식 프로세스는 Zombie 상태로 유지
- 메모리 낭비 (프로세스 테이블 항목)
- 시스템 리소스 누적
- PID 고갈 가능

해결책:
```bash
wait()     # 부모가 자식 종료 상태 회수
```

## 5️⃣ IPC (Inter-Process Communication): 5가지 방법

프로세스는 격리되어 있지만, 때로는 **통신**이 필요합니다.

### 1️⃣ Pipe (익명 파이프)

```
프로세스 A --[|]-- 프로세스 B
         (단방향)
```

**특징**:
- 부모-자식 프로세스 간만 사용 가능
- 단방향 통신
- 메모리 기반 (임시)

**예시**:
```bash
cat file.txt | grep "pattern" | wc -l
```

**커널 구현**:
- 내부 버퍼 (일반적으로 64KB)
- 쓰기 쪽이 가득 차면 읽을 때까지 블로킹

### 2️⃣ Named Pipe (FIFO)

```
파일 시스템에 존재하는 파이프
/tmp/mypipe (일반 파일처럼 보이지만 파이프)
```

**특징**:
- 이름이 있음 (파일 시스템에 존재)
- 관련 없는 프로세스 간 통신 가능
- 단방향

**예시**:
```bash
# 터미널 1
mkfifo /tmp/mypipe
cat /tmp/mypipe

# 터미널 2
echo "Hello" > /tmp/mypipe
# 터미널 1에 "Hello" 출력됨
```

### 3️⃣ Socket (가장 유연함)

```
프로세스 A ←→ Socket ←→ 프로세스 B (양방향)
           (로컬 또는 네트워크)
```

**특징**:
- 양방향 통신
- 로컬 머신 (Unix socket) 또는 네트워크 (TCP/UDP)
- 클라이언트-서버 모델

**예시**:
```bash
# 서버
nc -l 5000

# 클라이언트 (다른 터미널)
echo "Hello" | nc localhost 5000
```

### 4️⃣ Message Queue (System V IPC)

```
메시지 1 → [큐]
메시지 2 → [큐] ← 프로세스 B가 수신
메시지 3 → [큐]
```

**특징**:
- 메시지 타입으로 분류 가능
- 비동기 통신 (송신자가 수신자를 기다리지 않음)
- 커널이 메시지 저장

### 5️⃣ Shared Memory (가장 빠름)

```
메모리 영역 (공유)
├─ 프로세스 A (읽기/쓰기)
└─ 프로세스 B (읽기/쓰기)
```

**특징**:
- 직접 메모리 공유 (복사 없음)
- 가장 빠른 IPC
- **동기화 필요** (Semaphore)

**문제**:
```
프로세스 A: 메모리 값 = 10
프로세스 B: 메모리 값을 읽음 → 15를 봄 (뭔가 틀림)
프로세스 A: 동시에 값을 5로 변경 중
→ Race condition 발생!
```

## 6️⃣ 신호 (Signals): 프로세스 간 통신의 또 다른 형태

신호 = 비동기 이벤트 알림

### 주요 신호

| 신호 | 번호 | 의미 | 기본 동작 |
|------|------|------|---------|
| SIGTERM | 15 | 종료 요청 | 프로세스 종료 |
| SIGKILL | 9 | 강제 종료 | 프로세스 강제 종료 (무시 불가) |
| SIGSTOP | 19 | 중지 | 프로세스 일시 중지 |
| SIGCONT | 18 | 재개 | 중지된 프로세스 재개 |
| SIGUSR1 | 10 | 사용자 정의 | 프로그램이 정의 |

### Kubernetes와의 연결

```
Pod 종료 시:
1. kubelet이 Pod의 PID 1 프로세스에 SIGTERM 전송
2. 프로세스가 graceful shutdown 수행 (30초 기한)
3. 30초 후에도 종료 안 되면 SIGKILL 전송
4. 프로세스 강제 종료
```

## 7️⃣ 컨텍스트 스위칭: 멀티프로세싱의 비결

### 문제: CPU는 1개인데 프로세스는 많음?

```
프로세스 A, B, C, D, ... 가 모두 실행되고 있는 것처럼 보임
실제로는: 매우 빠르게 프로세스를 전환하면서 실행
```

### 해결책: 컨텍스트 스위칭

```
CPU에서 프로세스 A 실행
   ↓
인터럽트 발생 (타이머 또는 I/O)
   ↓
프로세스 A의 상태 저장 (레지스터, PC, 메모리 포인터)
   ↓
프로세스 B의 상태 복구
   ↓
CPU에서 프로세스 B 실행
   ↓
(반복)
```

### 성능 영향

```
스위칭이 너무 자주 발생 → 오버헤드 증가 → 성능 저하
→ 타임 슬라이스 (time quantum) 조정이 필요
```

## 8️⃣ Namespace: 격리의 진화 (다음 장 미리보기)

지금까지는 **프로세스 자체 격리**를 배웠습니다.

다음 단계: **네임스페이스** = 격리의 업그레이드

```
프로세스만 격리 (현재)
  ↓
커널 서브시스템도 격리 (다음 장)
  ├─ PID Namespace (프로세스 ID 공간)
  ├─ Network Namespace (네트워크 인터페이스)
  ├─ Mount Namespace (파일 시스템)
  ├─ UTS Namespace (호스트명)
  └─ IPC Namespace (IPC 리소스)
```

## 9️⃣ AKS에서의 프로세스

### AKS 노드의 프로세스 계층

```
kubelet (PID 1000)
├─ containerd (PID 1001)
│  ├─ runc (PID 1002) -- PID Namespace 생성 -- 
│  │                                          │
│  │                                          ↓
│  │                                    Pod의 PID 1
│  │                                    (app)
│  │
│  ├─ runc (PID 1003) --
│  │
│  └─ runc (PID 1004) --
│
└─ kube-proxy (PID 1100)
```

### 컨테이너 내부 프로세스 확인

```bash
# 호스트에서
ps aux | grep "app"
# 출력: PID 5000

# Pod 내부에서
kubectl exec -it <pod> -- ps aux
# 출력: PID 1 (완전히 다른 PID!)
```

**이유**: 각 Pod이 자신의 **PID Namespace**를 가짐
