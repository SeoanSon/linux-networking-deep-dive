## 📖 1. Routing이란?

### 정의
**Routing**은 **출발지에서 목적지까지 패킷을 전달하는 경로를 결정하는 프로세스**입니다.

```
라우팅의 핵심:
패킷이 들어옴 (Destination IP: 10.0.2.5)
         ↓
라우팅 테이블 조회
         ↓
"10.0.2.5는 어느 경로로 가지?"
         ↓
경로 결정: "eth0으로 전송"
         ↓
패킷 전송
```

### 라우팅 테이블

```
Linux 라우팅 테이블 예:
$ ip route show
10.244.0.0/24 dev cbr0 scope link src 10.244.0.1
10.244.1.0/24 via 10.240.0.6 dev eth0
10.244.2.0/24 via 10.240.0.7 dev eth0
172.17.0.0/16 dev docker0 proto kernel scope link
default via 10.240.0.1 dev eth0

각 행의 의미:
┌──────────────────────────────────────────┐
│ 10.244.0.0/24 dev cbr0 scope link        │
├──────────────────────────────────────────┤
│ Destination: 10.244.0.0/24               │ 목적지 네트워크
│ (CIDR 형태)                              │
│                                          │
│ Device: cbr0                             │ 어느 인터페이스로
│                                          │
│ Scope: link                              │ 직접 연결 (gateway 없음)
│                                          │
│ Source: 10.244.0.1                       │ 출발지 IP
└──────────────────────────────────────────┘

↓

┌──────────────────────────────────────────┐
│ 10.244.1.0/24 via 10.240.0.6 dev eth0   │
├──────────────────────────────────────────┤
│ Destination: 10.244.1.0/24               │ 목적지 네트워크
│                                          │
│ Gateway: 10.240.0.6                      │ 거쳐가야 할 IP
│                                          │
│ Device: eth0                             │ 어느 인터페이스로
│                                          │
│ (scope는 생략, 기본값: unicast)          │
└──────────────────────────────────────────┘
```

## 📖 2. Routing의 작동 원리

### 2.1 라우팅 결정 프로세스

```
패킷 도착: Destination IP = 10.244.1.5

1️⃣ 라우팅 테이블 스캔 (순서대로)
   
   10.244.0.0/24? 아니다 (10.244.1.5는 이 범위 아님)
   10.244.1.0/24? YES! (10.244.1.5는 이 범위임)
   
2️⃣ 매칭된 라우트 사용
   
   Destination: 10.244.1.0/24
   Gateway: 10.240.0.6 (다음 홉)
   Device: eth0

3️⃣ Gateway로 패킷 전송
   
   패킷의 다음 홉: 10.240.0.6
   출발 인터페이스: eth0
   
4️⃣ 구체적으로 어떻게?
   
   a) 다음 홉의 MAC 주소 필요
      └─ ARP: "누가 10.240.0.6인가?"
      └─ ARP 응답: MAC = 02:42:0a:f0:00:06
   
   b) Ethernet Frame 생성
      Src MAC: eth0의 MAC (02:42:0a:f0:00:01)
      Dst MAC: 10.240.0.6의 MAC (02:42:0a:f0:00:06)
      IP packet: (원본 그대로)
         Src IP: 10.244.0.5
         Dst IP: 10.244.1.5
   
   c) eth0으로 전송
```

### 2.2 CIDR와 매칭

```
라우팅에서 CIDR의 중요성:

테이블:
10.244.0.0/24 dev cbr0
10.0.0.0/8 via 10.240.0.1 dev eth0
default via 10.240.0.1

패킷: 10.244.0.100
     ├─ 10.244.0.0/24? YES (가장 구체적)
     └─ route: cbr0

패킷: 10.0.50.100
     ├─ 10.244.0.0/24? NO
     ├─ 10.0.0.0/8? YES (/8이 더 포괄적)
     └─ route: via 10.240.0.1

패킷: 8.8.8.8
     ├─ 10.244.0.0/24? NO
     ├─ 10.0.0.0/8? NO
     ├─ default? YES (가장 마지막 수단)
     └─ route: via 10.240.0.1

규칙: 가장 구체적인(longest prefix match) 라우트 사용
```

## 📖 3. Linux의 라우팅 구현

### 3.1 커널 라우팅 엔진

```
Linux Kernel Routing:

net/ipv4/route.c (메인 파일)

IP 패킷 도착:
    ↓
ip_route_input() 함수 호출
    ↓
라우팅 캐시 확인 (빠른 조회)
    ├─ 히트: 캐시된 라우트 사용 (빠름)
    └─ 미스: 라우팅 테이블 조회
         ↓
    fib_lookup() 호출 (FIB = Forwarding Information Base)
         ↓
    Radix tree 탐색 (구조화된 테이블)
         ↓
    매칭된 라우트 반환
         ↓
    라우팅 캐시에 저장 (다음번을 위해)
    ↓
다음 홉 결정 및 패킷 전송
```

### 3.2 라우팅 캐시

```
Linux 라우팅 성능 최적화:

처음 패킷:
10.244.1.5로 갈 10.244.1.0/24 라우트 찾기
└─ 라우팅 테이블 탐색 (시간 소요)
└─ 결과를 캐시에 저장

다음 패킷 (같은 목적지):
캐시 확인: "10.244.1.0/24? 있다!"
└─ 즉시 사용 (테이블 탐색 없음)

성능:
캐시 히트: ⭐⭐⭐⭐⭐ (매우 빠름)
캐시 미스: ⭐⭐⭐ (라우팅 테이블 탐색)

현대 커널:
└─ 라우팅 캐시 제거 (복잡도 증가로)
└─ Radix tree + 메모리 계층화로 최적화
```

## 📖 4. 라우팅의 종류

### 4.1 직접 연결 (Direct Connection)

```
라우팅 테이블:
10.244.0.0/24 dev cbr0 scope link

의미:
"10.244.0.0/24 네트워크는 cbr0에 직접 연결됨"

작동:
패킷 → 10.244.0.50
     └─ br0에 직접 전송
     └─ ARP로 MAC 찾기
     └─ 같은 네트워크 (L2 switching)
```

### 4.2 간접 연결 (Indirect via Gateway)

```
라우팅 테이블:
10.244.1.0/24 via 10.240.0.6 dev eth0

의미:
"10.244.1.0/24로 가려면 10.240.0.6을 거쳐야 함"

작동:
패킷 → 10.244.1.50 (최종 목적지)
     └─ 다음 홉: 10.240.0.6 (게이트웨이)
     └─ 10.240.0.6의 MAC 찾기 (ARP)
     └─ eth0으로 전송
     └─ 10.240.0.6이 다음 처리
```

### 4.3 기본 라우트 (Default Route)

```
라우팅 테이블:
default via 10.240.0.1 dev eth0

의미:
"어디로 가야 할지 모르면 10.240.0.1로 보내"

작동:
매칭되는 라우트 없음
└─ default route 사용
└─ 모든 패킷을 10.240.0.1로 전송
└─ 게이트웨이가 라우팅 결정
```

## 📖 5. Kubernetes와 라우팅

### 5.1 Kubernetes 라우팍 테이블

```
AKS Node의 라우팅 테이블:

10.244.0.0/24 dev cbr0 scope link  # 로컬 Pod들
10.244.1.0/24 via 10.240.0.6 dev eth0  # Node 2의 Pod들
10.244.2.0/24 via 10.240.0.7 dev eth0  # Node 3의 Pod들
...
0.0.0.0/0 via 10.240.0.1 dev eth0  # 외부 트래픽

각 행의 의미:
├─ 로컬 Pod: cbr0으로 직접
├─ 다른 Node의 Pod: 해당 Node IP를 통해
└─ 인터넷: 게이트웨이로
```

### 5.2 Pod 간 통신의 라우팅

```
Pod A (Node 1, 10.244.0.5)에서
Pod E (Node 2, 10.244.1.7)로 통신:

1️⃣ Pod A에서 패킷 생성
   Src IP: 10.244.0.5
   Dst IP: 10.244.1.7

2️⃣ Pod A의 라우팅 테이블 확인
   "10.244.1.7은 어디로 가지?"
   └─ 매칭: 10.244.1.0/24 via 10.240.0.6
   └─ 다음 홉: 10.240.0.6 (Node 2의 IP)

3️⃣ Node 1의 eth0으로 전송
   Dst MAC: 10.240.0.6의 MAC
   IP: (변경 없음, 원본 유지)

4️⃣ Node 2가 패킷 수신
   "이건 10.244.1.7이 목적지네"
   └─ Node 2의 라우팅 테이블 확인
   └─ 10.244.1.0/24 dev cbr0
   └─ cbr0으로 전송

5️⃣ cbr0이 Mac 학습 및 포워딩
   veth to Pod E

6️⃣ Pod E가 수신
```

## 📖 6. Routing의 설정

### 6.1 라우트 추가/삭제

```
라우트 추가:
$ ip route add 10.244.1.0/24 via 10.240.0.6 dev eth0

라우트 삭제:
$ ip route del 10.244.1.0/24 via 10.240.0.6

기본 게이트웨이 설정:
$ ip route add default via 10.240.0.1 dev eth0

확인:
$ ip route show
$ ip route show table all
```

### 6.2 Routing policy (PBR)

```
일반 라우팅:
목적지 IP만 봐서 경로 결정

Policy-based routing:
여러 기준으로 경로 결정:
├─ 출발지 IP
├─ 포트
├─ 프로토콜
├─ 사용자
└─ ToS (Type of Service)

예: Pod의 사용자가 "admin"이면 다른 경로로
$ ip rule add from 10.244.0.0/24 lookup 100
$ ip route add default via 10.240.0.1 table 100
```

## 📖 7. 라우팅의 성능

### 7.1 라우팅 오버헤드

```
패킷 처리 단계:
1️⃣ IP 헤더 검증: ~10 사이클
2️⃣ 라우팅 테이블 조회: ~100-500 사이클 (캐시히트시 훨씬 빠름)
3️⃣ TTL 감소 및 체크섬: ~20 사이클
4️⃣ Egress 포트 결정: ~50 사이클
─────────────────────────────────
합계: ~180-580 사이클 / 패킷

최신 CPU (3 GHz):
└─ 180 사이클 = 60ns (매우 빠름)
```

### 7.2 라우팅 최적화

```
최적화 방법:

1️⃣ 라우팅 캐시 활용
   └─ 같은 목적지 반복 = 캐시 히트
   └─ Long flow에서 효율적

2️⃣ Longest Prefix Match 최적화
   └─ Radix tree 대신 더 효율적인 자료구조
   └─ Trie, Hash table 등

3️⃣ ECMP (Equal-Cost Multi-Path)
   └─ 같은 비용의 여러 경로 사용
   └─ 부하 분산

예: 두 개의 gateway로 분산
10.0.0.0/8 via 10.240.0.1 dev eth0
10.0.0.0/8 via 10.240.0.2 dev eth0  # ECMP
└─ 플로우를 두 경로에 분산
```
